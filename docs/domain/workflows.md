# Domain workflows (end-to-end)

## 1) Создание клиента и сделки

1. UI: `/clients` -> создать клиента.
2. API: `POST /api/v1/clients/`.
3. UI: `/deals` -> создать сделку.
4. API: `POST /api/v1/deals/`.

## 2) Расчет и оформление полиса

1. UI: в сделке создать расчет.
2. API: `POST /api/v1/quotes/`.
3. UI: создать полис из сделки; для КАСКО можно указать франшизу, официального дилера и риск GAP. Продление отмечается отдельно во вкладке полисов сделки.
4. API формы: `POST /api/v1/policies/draft/` создаёт полис, платежи и финзаписи атомарно; `PATCH /api/v1/policies/<id>/draft/` обновляет этот же draft-граф. Старые `/policies/` CRUD endpoints остаются для простых/legacy изменений. Признак продления меняется через `PATCH /api/v1/policies/<id>/` с `is_renewed`.
5. Перенос полиса в другую сделку: UI выбирает целевую сделку, API вызывает `POST /api/v1/policies/<id>/move/` с `deal`; backend сначала переносит/создает папку полиса в Google Drive, затем в одной транзакции меняет `Policy.deal` и синхронизирует `Payment.deal`.

## 3) Платежи и финансовые записи

1. UI формы полиса отправляет платежи и финзаписи вложенно через policy draft endpoint.
2. API draft-сохранения держит `Payment.deal` синхронизированным с `Policy.deal`, нормализует знак доходов/расходов и применяет правила ведомостей в одной транзакции.
3. Отдельные endpoints `POST /api/v1/payments/` и `POST /api/v1/financial_records/` остаются для совместимости и точечных операций.

## 4) Ведомости (income/expense)

1. UI: `/commissions` -> создать ведомость.
2. API: `POST /api/v1/finance_statements/`.
3. Добавить/удалить записи в ведомости.
4. API: `PATCH /api/v1/finance_statements/<id>/`.

## 5) Задачи по сделке

1. UI: `/deals` -> добавить задачу.
2. API: `POST /api/v1/tasks/`.
3. Изменение статуса/сроков.
4. API: `PATCH /api/v1/tasks/<id>/`.
5. При отметке задачи выполненной можно указать комментарий выполнения; он сохраняется в задаче, отображается в таблице и добавляется в Telegram-уведомление.

## 6) Документы и распознавание

1. UI: загрузка документа в сделке.
2. API: `POST /api/v1/documents/`.
3. Запуск распознавания.
4. API: `POST /api/v1/documents/recognize/` (JWT required).

## 7) Knowledge / Open Notebook

1. UI: `/knowledge` -> работа с источниками/ноутбуками.
2. API: `GET/POST /api/v1/knowledge/sources/` и `/api/v1/knowledge/notebooks/`.
3. Диалог: `POST /api/v1/knowledge/ask/` (JWT required).

## 8) Чат по сделке

1. UI: вкладка чата в сделке.
2. API: `GET /api/v1/chat_messages/?deal=<id>`.
3. Отправка сообщения: `POST /api/v1/chat_messages/`.

## 9) Уведомления и Telegram

1. UI: `/settings` -> настройка уведомлений.
2. API: `GET/PUT /api/v1/notifications/settings/`.
3. Привязка Telegram: `POST /api/v1/notifications/telegram-link/`.
4. Пользователь открывает ссылку/бота и отправляет `/start <код>`.
5. Bot: сохраняет `TelegramProfile.chat_id`.
6. Backend: отправляет пользователю Telegram-уведомления о задачах, сделках, оплатах и полисах.
7. Работа со сделками и документами через Telegram-бота отключена; файлы загружаются через UI/API CRM.

## 10) Аутентификация

1. UI: login.
2. API: `POST /api/v1/auth/login/` -> JWT.
3. Обновление токена: `POST /api/v1/auth/refresh/`.

## 11) Закрытие сделки

1. UI: карточка сделки -> закрыть (выбор `won`/`lost`, указать причину).
2. API: `POST /api/v1/deals/<id>/close/`.
3. Backend: валидация статуса (`won`/`lost`) и сохранение `closing_reason`.

## 12) Лента событий сделки и причина даты

1. UI: рядом с read-only полем сделки `Крайний срок` показывает компактный блок `Почему эта дата`: точные причины по совпадающей дате, предупреждение о рассинхроне или понятное пустое состояние.
2. API: `GET /api/v1/deals/<id>/events/` возвращает человекочитаемую бизнес-ленту событий сделки; технический audit log остается отдельным endpoint `history`.
3. Backend: `Deal.expected_close` вычисляется как ближайшая дата из `manual_expected_close`, непродлённых полисов и неоплаченных платежей; прямой PATCH `expected_close` запрещён.
4. Backend: ручной крайний срок добавляется через событие `manual_expected_close`; `next_contact_date` остается напоминанием и не сохраняется в бизнес-ленту `DealEvent`.
5. UI: кнопка рядом с `Крайний срок` открывает модальное окно ручного крайнего срока формата `дата + причина`; отдельного типа обычного ручного события в пользовательском интерфейсе нет.
6. Backend: старые ручные дедлайны, перенесенные из `expected_close`, и прежние события `manual` материализуются/конвертируются в ленту как `manual_expected_close`, чтобы `Почему эта дата` не показывал рассинхрон.
7. UI: события в `Ленте` сортируются по близости к текущей дате; прошедшие события приглушаются, а будущие используют цветовую срочность как `Крайний срок` в списке сделок.
8. UI: вкладка `Журнал` показывает техническую историю изменений из `AuditLog` и не смешивается с бизнес-лентой.

## 13) Legacy-статус и продление полиса

1. `Policy.status` остаётся legacy-полем БД/старого CRUD API.
2. Новые UI-потоки не редактируют `status`; отображение строится от computed status и финансовых признаков.
3. Продление хранится в `Policy.is_renewed` и меняется lightweight patch через `PATCH /api/v1/policies/<id>/`.

## 14) Ручная обработка дублей клиентов

1. UI: списки клиентов, сделок и полисов показывают тихий индикатор рядом с ФИО, если для клиента есть кандидаты на слияние или ФИО можно безопасно нормализовать.
2. API: `POST /api/v1/clients/duplicate-hints/` принимает `client_ids` и возвращает подсказки пачкой: количество кандидатов, максимальный score/confidence, причины совпадения и preview нормализованного ФИО.
3. UI: клик по индикатору дублей открывает ручной просмотр кандидатов; кнопка `Объединить` у кандидата сразу открывает merge preview для выбранной пары, а `Это разные` сохраняет глобальное исключение пары и убирает ложный дубль из подсказок. Автоматического объединения нет.
4. Merge доступен любому авторизованному пользователю CRM; редактирование и удаление клиентов вне merge остаются ограничены владельцем или администратором.
5. UI: подтвержденный merge выполняется пошагово через backend-сессию: документы Drive переносятся короткими запросами с progress bar и retry для временных ошибок, а изменения в БД применяются только на финальном шаге.
6. API: `POST /api/v1/clients/similar/` возвращает кандидатов с причинами, score и счетчиками связанных сделок/полисов; `POST /api/v1/clients/similarity-exclusions/` принимает `target_client_id` + `candidate_client_id` и исключает эту симметричную пару из будущих результатов.
7. Preview слияния по умолчанию предлагает самое длинное ФИО, оставляет телефон/email целевого клиента основными, но если они пустые, подставляет первый непустой контакт из дублей. Остальные альтернативные контакты источников добавляются в итоговые заметки. Разные форматы одного телефона считаются одним номером.
8. UI: клик по индикатору нормализации показывает подтверждение `было -> станет`.
9. API: `POST /api/v1/clients/<id>/normalize-name/` меняет только `name`, сохраняет audit log и не запускается автоматически при обычном сохранении клиента.
