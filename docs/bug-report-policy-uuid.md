# Отчет об ошибке: при добавлении полиса отображается `Must be a valid UUID`

Дата диагностики: 17.06.2026  
Окружение: production CRM 3.0  
Затронутый сценарий: добавление полиса в сделку из вкладки "Полисы"

## Кратко

При попытке добавить полис `SYS2851082838` в сделку, которая в интерфейсе отображается как `#16c3bcb8`, пользователь получает ошибку `Must be a valid UUID`.

Серверная логика создания полиса работает, если передать полный UUID сделки и UUID справочников. Полис не появился в интерфейсе после диагностики, потому что проверка выполнялась в транзакции с принудительным rollback, чтобы не изменять production-данные.

## Данные кейса

- Файл: `Polis CASCO SYS2851082838.pdf`
- Сделка в UI: `#16c3bcb8`
- Полный UUID сделки на сервере: `16c3bcb8-4118-433b-81aa-5a6385538ece`
- Название сделки: `VOLKSWAGEN TIGUAN 2019 M361OY799`
- Номер полиса: `SYS2851082838`
- Страховая компания: `РЕСО-ГАРАНТИЯ`
- UUID страховой компании: `d2be7261-c17e-441c-9214-054060e288cf`
- Тип страхования: `КАСКО`
- UUID типа страхования: `a3f5b7fa-787c-413b-837f-00a8b32f2d43`
- VIN: `XW8ZZZ5NZKG236893`
- Срок действия: `26.07.2025` - `25.07.2026`
- Премия: `44028.00`
- Клиент сделки: `8ca0a33d-5b11-4b5b-9617-0964cab6a9df`

## Что проверено

1. На production проверено, что полис `SYS2851082838` отсутствует в базе до теста.
2. Через Django `PolicyDraftSerializer` и сервис `apply_policy_draft()` выполнен dry-run создания полиса.
3. Dry-run выполнялся внутри `transaction.atomic()` с `transaction.set_rollback(True)`.
4. С корректными UUID serializer прошел валидацию, сервис создал бы полис и один платеж.
5. После rollback полис `SYS2851082838` по-прежнему отсутствует в базе.

Фактический результат dry-run:

```text
before_count 0
serializer_valid True
serializer_errors {}
created_policy_id 07cede94-f524-418f-b504-34d4e0ae5ebf
created_policy_number SYS2851082838
created_client_id 8ca0a33d-5b11-4b5b-9617-0964cab6a9df
payments_count 1
after_count 0
```

## Почему пользователь не видит полис

Полис не был сохранен в production-базу. Диагностика намеренно выполнялась как безопасный dry-run с rollback, потому что исходное ограничение было: "Код на сервере не трогай! Только анализ!".

`created_policy_id` в выводе - это временный объект, созданный внутри транзакции. После rollback он удален, поэтому в UI счетчик полисов остается `0`.

## Воспроизведение ошибки

На production serializer возвращает UUID-ошибку, если вместо полного UUID передать короткий id сделки или текстовые значения справочников:

```text
case deal '16c3bcb8'
valid False
errors {'deal': ['"16c3bcb8" is not a valid UUID.']}

case insurance_company 'РЕСО-ГАРАНТИЯ'
valid False
errors {'insurance_company': ['"РЕСО-ГАРАНТИЯ" is not a valid UUID.']}

case insurance_type 'КАСКО'
valid False
errors {'insurance_type': ['"КАСКО" is not a valid UUID.']}
```

## Вероятная причина

В запрос на `/api/v1/policies/draft/` уходит не полный UUID для одного из полей, которые backend ожидает как `PrimaryKeyRelatedField`.

Наиболее вероятное поле: `deal`.

Причина: в UI сделка отображается коротким идентификатором `#16c3bcb8`. Если этот короткий id используется при отправке формы вместо полного `16c3bcb8-4118-433b-81aa-5a6385538ece`, backend закономерно возвращает `Must be a valid UUID`.

Также аналогичная ошибка возможна, если frontend отправляет:

- `insurance_company: "РЕСО-ГАРАНТИЯ"` вместо UUID компании;
- `insurance_type: "КАСКО"` вместо UUID типа;
- `sales_channel` как название канала вместо UUID;
- `client` как имя клиента вместо UUID.

## Ожидаемое поведение

Frontend должен отправлять в payload только полные UUID для связанных объектов:

```json
{
  "deal": "16c3bcb8-4118-433b-81aa-5a6385538ece",
  "insurance_company": "d2be7261-c17e-441c-9214-054060e288cf",
  "insurance_type": "a3f5b7fa-787c-413b-837f-00a8b32f2d43",
  "client": "8ca0a33d-5b11-4b5b-9617-0964cab6a9df",
  "number": "SYS2851082838"
}
```

Если пользователь вводит нового страхователя текстом, frontend может передавать `client_name`, но поле `client` при этом должно быть пустым или валидным UUID.

## Рекомендации для исправления

1. Проверить payload в браузере DevTools при сохранении полиса.
2. Убедиться, что `dealId`, передаваемый в `handleAddPolicy(dealId, values)`, является полным UUID, а не коротким отображаемым id.
3. Проверить сценарий открытия модалки создания полиса из вкладки сделки: UI может показывать короткий id `#16c3bcb8`, но состояние формы/API должно хранить полный `deal.id`.
4. Проверить маппинг результата распознавания PDF: названия страховой и типа должны матчиться на справочники и превращаться в UUID до отправки.
5. Улучшить отображение ошибок API на frontend: вместо общего `Must be a valid UUID` показывать поле, например `Некорректный идентификатор сделки` или `Не выбрана страховая компания`.
6. Добавить frontend-тест на создание полиса из карточки сделки, где UI показывает короткий id, но API получает полный UUID.

## Риск

Backend в этом сценарии выглядит корректным: он ожидает UUID и валидно отвергает короткие/текстовые значения. Основной риск находится в frontend-состоянии или в маппинге данных распознавания PDF перед отправкой формы.

## Статус

Код и production-данные не изменялись. Полис можно создать серверными методами, если использовать полный UUID сделки и корректные UUID справочников.
