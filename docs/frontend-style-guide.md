# Frontend Style Guide

Этот документ фиксирует обязательный инженерный стандарт для `frontend/`.
Цель: снизить разнобой в React/TypeScript-коде, упростить ревью и сделать переиспользование UI-примитивов правилом, а не опцией.

## Основные правила

### 1. Компоненты и хуки
- Компоненты объявляем именованными функциями: `export function Component(props: Props) {}`.
- Хуки объявляем именованными функциями: `export function useThing(...) {}`.
- `React.FC` и `React.FunctionComponent` не используем.
- `default export` не используем, кроме файлов tooling-конфигурации, где это требует сам инструмент.

### 2. Типы и контракты
- Для props и object-shaped контрактов используем `interface`.
- `type` оставляем для union-типов, mapped types, utility aliases и других композиций, где `interface` неудобен.
- Где возможно, используем `import type`, чтобы отделять типы от runtime-импортов.

### 3. Структура компонентов
- Крупный экран не должен одновременно держать orchestration, layout и повторяющиеся UI-рецепты.
- Если в компоненте появляются повторяющиеся JSX-блоки или наборы className, выносим их:
  - в `frontend/src/components/common/*` для общих примитивов,
  - в локальные helper-компоненты рядом с feature-компонентом, если паттерн специфичен для одного экрана.
- Feature-компоненты должны собираться из секций, helper-функций и общих примитивов, а не из длинных inline-разметок.

### 4. Общие style primitives
- Кнопки, панели, таблицы, поля ввода, muted/alert-состояния и modal-shells сначала ищем в `frontend/src/components/common/*`.
- Повторяющиеся UI-строки классов не держим внутри feature-компонентов, если они уже встречаются больше одного раза.
- При добавлении нового визуального паттерна сначала решаем, общий он или локальный:
  - общий паттерн выносим в `common/buttonStyles.ts`, `common/uiClassNames.ts`, `common/tableStyles.ts` или соответствующий primitive;
  - локальный паттерн оставляем рядом с экраном, но оформляем как именованный helper, а не как анонимную строку внутри JSX.

### 5. Именование и экспорт
- Используем named exports only.
- Имена компонентов, хуков и helpers должны отражать ответственность, а не только визуальную форму.
- Пропсы называем `<ComponentName>Props`, локальные вспомогательные интерфейсы `...Props` или `...State`, если это действительно контракты.

## Что проверяется автоматически
- `npm run lint` запрещает:
  - `React.FC`,
  - `React.FunctionComponent`,
  - `default export` во frontend-коде,
  - `type` для object-shaped контрактов вместо `interface`.

## Правила внедрения
- Любой новый frontend-код сразу должен соответствовать этому гайду.
- Любой затронутый существующий файл приводим к стандарту в рамках той же задачи.
- Массовый косметический рефакторинг без причины не нужен, но оставлять новые расхождения в уже изменяемом файле нельзя.

## Мини-чеклист перед PR
- Компонент объявлен как именованная функция.
- Props описаны через `interface`.
- Нет `default export`.
- Повторяющиеся className не размножаются в feature-компоненте без причины.
- Общие кнопки, панели, таблицы и поля собраны из shared primitives.
- `npm run lint`, `npm run format:check`, `npm run test`, `npm run build` проходят.
